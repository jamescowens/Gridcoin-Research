## Relationship to Section 1

Section 1 (Owens, ILikeChocolate, Held, 2018) models staking as a series of
independent Bernoulli trials with probability p_i per UTXO per 16-second
quantum, and derives ETTS from the resulting geometric/Poisson distributions.
It explicitly treats the stake hash as "a random 256 bit number" and defers
the cryptographic details to "a future document."

This paper is that future document. It opens the black box of the kernel
function, explains why its deterministic output behaves as a random oracle,
and provides a security analysis of each input against manipulation. The
notation from Section 1 (T, T_U, D, H_max, w_i, v_i, p_i, etc.) is reused
throughout.

## Outline

### 1. Introduction

- Recap: Section 1 established the probabilistic framework for staking
  - Bernoulli trials at 16-second intervals (Section 1, §1)
  - p_i = T * w_i / H_max (Section 1, Eqn. 2)
  - The hash was treated as uniformly random over [0, 2^256 - 1]
- This paper examines the mechanism that produces the hash
  - The kernel is a fully deterministic function: given the same five
    inputs, it always produces the same 256-bit output
  - It is a seeded pseudorandom number generator — the stake modifier
    acts as the seed, and the UTXO identity + time provide the counter
  - The "randomness" is a consequence of SHA-256d's properties, not any
    physical source of entropy
- Goal: describe the kernel's structure, prove it generates outputs
  consistent with the random oracle assumption underlying Section 1's
  probability model, and analyze each input for resistance to manipulation

### 2. The Kernel Function

#### 2.1. Hash Construction

The V8+ kernel hash K is computed as:

    K = SHA256d( M || mask(t_b) || h_tx || n || mask(t_s) )

where:
- M: 64-bit stake modifier (from chain history; see §3)
- t_b: block timestamp of the UTXO's confirming block
- h_tx: transaction hash (SHA256d) of the UTXO's transaction
- n: output index of the UTXO within that transaction
- t_s: current (proposed stake) timestamp
- mask(t) = t AND NOT 0x0F  (zeroes the bottom 4 bits → 16-second quanta)

The preimage is 52 bytes: 8 + 4 + 32 + 4 + 4.

Source: `CalculateStakeHashV8()` in `src/gridcoin/staking/kernel.cpp`

#### 2.2. The Weighted Target Comparison

The staking trial succeeds when:

    uint320(K) <= uint320(T) * uint320(w_i)

where T is the 256-bit unweighted target (from nBits compact encoding)
and w_i = floor(v_i / 0.0125) = floor(80 * v_i) is the UTXO weight, using
the same definition as Section 1, Eqn. 2.

Why 320 bits: T * w_i can exceed 2^256 for large UTXOs. The 320-bit
comparison avoids overflow while maintaining exact arithmetic.

This inequality is the "coin toss" described in Section 1, §1. The
probability that K satisfies it is exactly p_i = T * w_i / 2^256 from
Section 1, Eqn. 2 — provided K is uniformly distributed, which we justify
in §2.3.

Source: `CheckProofOfStakeV8()` in `src/gridcoin/staking/kernel.cpp`

#### 2.3. SHA-256d as a Random Oracle

- In cryptographic analysis, SHA-256 is modeled as a random oracle: for
  distinct inputs, the outputs are independently and uniformly distributed
  over {0,1}^256
- SHA-256d (double application) inherits this property
- The kernel inputs vary in at least one of: stake modifier (changes every
  ~10 min), masked block time (fixed per UTXO), tx hash (unique per tx),
  output index, or masked stake time (changes every 16s)
- Therefore each (UTXO, quantum) pair produces a distinct preimage and
  thus an effectively independent, uniform random 256-bit value
- This is the precise justification for Section 1's assumption that the
  hash "can be treated as a random 256 bit number with values from 0 to
  2^256 - 1"

### 3. The Stake Modifier

#### 3.1. Purpose

The stake modifier exists to prevent a UTXO creator from predicting the
kernel hash at the time of transaction creation. Without it, an attacker
could construct transactions whose outputs are guaranteed to stake at a
chosen future time.

(Quote from the source code: "The purpose of stake modifier is to prevent
a txout (coin) owner from computing future proof-of-stake generated by
this txout at the time of transaction confirmation.")

#### 3.2. Construction

- A new 64-bit modifier is generated every 10 minutes (nModifierInterval
  = 600 seconds)
- The modifier is assembled bit-by-bit from 64 blocks selected from a
  candidate window spanning the selection interval
- Each selected block contributes its "entropy bit" (the LSB of its block
  hash) to one bit position of the modifier
- Block selection is deterministic given the chain state: it depends on
  block timestamps and the previous modifier via a selection hash
- Candidate blocks are shuffled before sorting to break ties

Source: `ComputeNextStakeModifier()` in `src/gridcoin/staking/kernel.cpp`

#### 3.3. The Maturation Firewall

- `nStakeMinAge` = 16 hours (57,600 seconds) on mainnet; 1 hour on testnet
- During the 16-hour maturation period, approximately 96 new modifiers are
  generated (16h * 60min/h / 10min = 96)
- Each modifier depends on blocks that have not yet been produced at the
  time the UTXO is created
- Therefore the modifier that will apply when the UTXO becomes eligible to
  stake is unpredictable at creation time
- This is the fundamental security property: even though the kernel is
  deterministic, the seed (modifier) is unknown at UTXO creation

#### 3.4. Entropy Analysis

- 64 bits of modifier, with each bit contributed by an independent block
- An attacker controlling fraction f of the network's stake weight
  controls roughly f of the selected blocks
- To control all 64 bits, the attacker would need to produce all 64
  contributing blocks — requiring sustained supermajority stake
- Even partial control yields diminishing returns: controlling k of 64
  bits reduces the effective hash space by only a factor of 2^k, while
  requiring extraordinary sustained stake dominance

#### 3.5. Historical Note: The Modifier Reset

- A bug inherited from the grandfather rule corrupted the stake modifier
  chain on mainnet
- At height 1,009,994, the modifier was reset to a known-good value
  (0xdf209a3032807577) via a hardcoded correction in the source
- This was a one-time fix; the modifier computation has been correct since

### 4. Security Analysis of Kernel Inputs

This section analyzes each kernel input for susceptibility to manipulation.

#### 4.1. Stake Modifier (M) — Not Controllable

Covered in §3. The attacker cannot predict M at UTXO creation time, and
cannot retroactively change it at staking time (it is determined by the
chain state at the most recent modifier-generation boundary before the
staking block).

#### 4.2. Masked Block Time (mask(t_b)) — Fixed at Confirmation

- Set when the UTXO's transaction is confirmed in a block
- The UTXO creator does not choose which block confirms their transaction
- Even if they could influence confirmation timing, the 16-second mask
  limits the space to ~5,400 values/day
- Once confirmed, this value is immutable

#### 4.3. Transaction Hash (h_tx) — Grindable but Futile

- The UTXO creator controls the transaction structure and can therefore
  influence h_tx by varying inputs, outputs, or nLockTime
- This is the "grinding" attack addressed by Tomas Brod in the V8 design
- The defense: h_tx interacts with M in the kernel hash, and M is unknown
  at the time h_tx is chosen (§3.3)
- Grinding h_tx without knowing the future M is searching for a preimage
  of an unknown function — equivalent to random guessing
- Key quote from the source: "trying to brute-force good tx hash is not
  possible as it is not known what stake modifier will be after the coins
  mature"

#### 4.4. Output Index (n) — Minimal Control

- Determined by the transaction structure
- A transaction with k outputs provides k choices for n
- Combined with h_tx, this gives k * (grinding attempts) preimages
- Still defeated by the unknown modifier argument

#### 4.5. Masked Stake Time (mask(t_s)) — The Only Runtime Input

- The staker chooses t_s (the block timestamp), subject to:
  - Must be greater than the previous block's timestamp
  - Must be within the network's time tolerance (±2 hours)
  - Masked to 16-second quanta: only ~5,400 distinct values per day
- Since v12, the mask is enforced at consensus (not just locally):
  validators reject blocks where tx.nTime != mask(tx.nTime)
- Trying more quanta means waiting more real time, which is exactly what
  honest staking does — there is no shortcut

#### 4.6. Combined Security Argument

The kernel is secure because:
1. The only high-entropy input (M) is unpredictable at UTXO creation
2. The only runtime-controllable input (t_s) has very low entropy
   (effectively 1 choice per 16 seconds)
3. SHA-256d provides no structural shortcut for finding preimages
   that satisfy the target inequality
4. The net effect: each (UTXO, quantum) pair is an independent trial
   with probability p_i, exactly as assumed by Section 1

### 5. Time Quantization and the Staking Loop

#### 5.1. The 16-Second Mask

- STAKE_TIMESTAMP_MASK = 0x0F
- mask(t) zeroes the bottom 4 bits of the unix timestamp
- Within any 16-second quantum, the kernel hash for a given UTXO is
  invariant: the same five inputs produce the same hash
- This makes the system a discrete-time process with 5,400 steps per day

#### 5.2. Practical Loop Cadence

- The wallet's staking loop: traverse all eligible UTXOs, then sleep for
  `nMinerSleep` (default 8,000 ms, configurable via `-minersleep`)
- Total loop time = UTXO traversal time + 8 seconds
- For most wallet/hardware combinations, loop time < 16 seconds, so the
  loop completes twice per quantum
- The second traversal within the same quantum is degenerate: same masked
  time → same kernel hash → same (failed) result
- Net effect: one independent trial per UTXO per 16-second quantum, as
  assumed by Section 1

#### 5.3. Staking Efficiency

- The wallet tracks the fraction of quanta in which at least one kernel
  evaluation occurred (staking efficiency)
- If the UTXO traversal + sleep exceeds 16 seconds (very large UTXO sets,
  slow hardware), some quanta are missed → efficiency < 100%
- This affects the effective trial rate but not the per-trial probability

### 6. From Deterministic Trials to Poisson Arrival

This section formally bridges the kernel's deterministic structure to the
probabilistic model of Section 1.

#### 6.1. The Bernoulli Trial

For a single UTXO i in quantum q:
- The kernel hash K_iq is a deterministic function of (M_q, t_b, h_tx, n, t_s_q)
- Under the random oracle model (§2.3), K_iq is uniformly distributed
- P(K_iq <= T * w_i) = T * w_i / 2^256 = p_i (Section 1, Eqn. 2)
- The trial is independent of all other (UTXO, quantum) pairs because the
  preimages are distinct and SHA-256d outputs are uncorrelated

#### 6.2. The Poisson Limit

- For a wallet with n UTXOs, there are n trials per quantum
- The per-quantum probability of at least one success is:
  P_quantum = 1 - Π(1 - p_i) ≈ Σp_i = p_t (for small p_i)
  which is Section 1, Eqn. 4
- Over N quanta: the number of successes follows Binomial(N, p_t)
- Since p_t is extremely small and N is large, the Poisson limit theorem
  gives convergence to Poisson(λ) with λ = N * p_t
- Setting λ = 1 and solving for N gives the expected quanta to first
  success: N = 1/p_t, recovering Section 1, Eqn. 9

#### 6.3. Network Aggregation and Block Arrival

- Each wallet's staking process is an independent Poisson process with its
  own rate parameter
- The superposition of independent Poisson processes is Poisson with
  rate = sum of individual rates
- The network block arrival process is therefore Poisson with rate
  λ_net = (1/16s) * Σ_all_UTXOs p_i
- The difficulty retargeting algorithm (Section 1, §2.3, Eqn. 14) adjusts
  T to maintain λ_net ≈ 1/90s (one block per 90 seconds)
- This feedback loop is why the Poisson model holds even as the set of
  active stakers changes

#### 6.4. Independence: What Could Break It?

For the Poisson model to hold, successive trials must be independent. This
requires:
- Different quanta → different masked stake time → different preimage → OK
- Different UTXOs → different (h_tx, n) pair → different preimage → OK
- Same UTXO across modifier boundaries → different M → different preimage → OK
- Potential concern: two quanta within the same 10-minute modifier period
  share the same M, and the only varying input is mask(t_s). This is fine
  because SHA-256d produces uncorrelated outputs for inputs differing in
  even a single bit. The independence holds.

### 7. Quantitative Worked Example

#### 7.1. Kernel-Level Probability

Walk through a concrete example connecting the kernel to Section 1's ETTS:

- Given: 10,000 GRC UTXO, network difficulty D = 5
- w_i = floor(80 * 10,000) = 800,000
- T = T_U / D = (2^236 - 2^220) / 5 [expand the compact notation]
- p_i = T * w_i / 2^256 [compute numerically]
- E(trials) = 1 / p_i
- ETTS = 16s * E(trials) / 86,400 s/day
- Show this equals G * D / v_i from Section 1, Eqn. 13

#### 7.2. The 320-Bit Comparison in Practice

- For the 10,000 GRC UTXO: T * w_i < 2^256, so 256-bit arithmetic would
  suffice
- For a 1,000,000 GRC UTXO: T * w_i could exceed 2^256, hence the need
  for 320-bit arithmetic
- The 320-bit comparison preserves linearity of p_i with respect to w_i
  across all UTXO sizes

### 8. Historical Context

#### 8.1. Pre-V8 Kernel (PoS v3 Legacy)

- Kernel included RSA weight and magnitude in the hash inputs
- Tightly coupled staking probability to BOINC participation
- More complex input set made analysis difficult
- Grandfather rule relaxed validation for blocks below height 1,034,700
  (mainnet), creating the modifier corruption that required the reset

#### 8.2. V8 Kernel Design (Tomas Brod, 2017)

- Removed RSA weight and magnitude from the kernel
- Simplified to pure coin-weight PoS: weight proportional only to value
- Added stake modifier (adapted from Peercoin/Blackcoin lineage)
- Retained tx hash despite ppcoin developers' advisory against it,
  justified by the maturation-modifier argument (§4.3)

#### 8.3. Post-V8 Refinements

- V12 (block version 12): consensus enforcement of timestamp mask —
  validators now reject blocks with unmasked coinstake timestamps
- Stake modifier reset at height 1,009,994

### 9. Comparison with Other PoS Systems

Brief positioning of Gridcoin's kernel in the PoS landscape:

| System    | Kernel type              | Leader election      | Key difference |
|-----------|--------------------------|----------------------|----------------|
| Peercoin  | Hash-based (coin-age)    | Probabilistic        | Uses coin-age weight instead of raw value |
| Blackcoin | Hash-based (PoSv2/v3)   | Probabilistic        | Gridcoin's direct ancestor; no research layer |
| NXT       | Hash-based (account)     | Probabilistic        | Account-based, not UTXO-based |
| Ethereum  | BFT (Casper)             | Deterministic roster | Fundamentally different: committee-based, not hash lottery |
| Gridcoin  | Hash-based (PoSv2 + modifier) | Probabilistic   | Unique: research reward layer, modifier firewall |

All hash-based PoS systems share the same core design: a deterministic hash
function whose output is compared against a stake-weighted target. They
differ in input selection, weight computation, and the mechanism that
prevents precomputation (modifier vs. coin-age vs. account state).

### 10. Conclusion

- The staking kernel is a deterministic pseudorandom function, not a
  source of true randomness
- Its security rests on three pillars:
  1. The stake modifier is unpredictable at UTXO creation time
  2. SHA-256d is computationally indistinguishable from a random oracle
  3. The 16-hour maturation window spans ~96 modifier recomputations
- The Poisson model from Section 1 emerges naturally:
  - Each (UTXO, quantum) pair is an independent Bernoulli trial (§6.1)
  - The Poisson limit theorem applies because p_i is small and trials
    are numerous (§6.2)
  - Network aggregation preserves the Poisson property (§6.3)
- The kernel's deterministic nature is not a weakness — it is a strength.
  It enables exact verification by every node in the network, while the
  random oracle property ensures that no participant can predict or
  manipulate outcomes beyond their proportional stake

### B. Full Kernel Pseudocode

```
function EvaluateKernel(utxo, current_time, stake_modifier, nBits):
    // Quantize timestamps to 16-second boundaries
    masked_block_time = utxo.block_time AND NOT 0x0F
    masked_stake_time = current_time AND NOT 0x0F

    // Compute kernel hash (52-byte preimage)
    kernel_hash = SHA256d(
        stake_modifier      ||    // 8 bytes (uint64, little-endian)
        masked_block_time   ||    // 4 bytes (uint32)
        utxo.tx_hash        ||    // 32 bytes (uint256)
        utxo.output_index   ||    // 4 bytes (unsigned int)
        masked_stake_time         // 4 bytes (uint32)
    )

    // Compute weight (integer division; < 0.0125 GRC → weight 0)
    weight = utxo.value_halfords / 1,250,000

    // Expand compact target and compare in 320-bit space
    target = expand_compact(nBits)           // 256-bit
    weighted_target = uint320(target) * uint320(weight)

    return uint320(kernel_hash) <= weighted_target
```

### C. Stake Modifier Computation Detail

```
function ComputeNextStakeModifier(pindexPrev):
    // Only recompute every nModifierInterval (600 seconds)
    if current_modifier_age < nModifierInterval:
        return previous_modifier

    // Collect candidate blocks from the selection window
    selection_interval = sum of 64 section durations
    candidates = blocks in [pindexPrev.time - selection_interval, pindexPrev.time]

    // Shuffle to break timestamp ties, then sort by timestamp
    shuffle(candidates)
    sort(candidates, by=timestamp)

    // Select 64 blocks, one per round
    new_modifier = 0
    for round in 0..63:
        block = select_block(candidates, round, previous_modifier)
        entropy_bit = block.hash & 1    // LSB of block hash
        new_modifier |= (entropy_bit << round)

    return new_modifier
```
